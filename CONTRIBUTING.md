# Contributing to Roaree Benchmark Club

Thank you for your interest in contributing to the Roaree Benchmark Club! This document outlines our contribution guidelines and membership requirements.

## üîí Membership Requirement

**This is an invite-only repository.** Only vetted members may contribute. If you are not yet a member:

1. **Columbia Students:** Contact the repository maintainers through official Columbia University channels
2. **External Collaborators:** Must be sponsored by a current member and approved by the core team
3. Provide your background, research interests, and relevant experience in LLM evaluation

## üéì Vetting Process

All prospective members go through a vetting process:

1. **Initial Contact:** Reach out with your Columbia credentials or sponsorship
2. **Background Review:** The core team reviews your academic background and research interests
3. **Interview (if applicable):** Some candidates may be invited for a discussion
4. **Approval:** The core team votes on new member applications
5. **Onboarding:** Approved members receive repository access and orientation

**Timeline:** The vetting process typically takes 1-2 weeks.

## üíª Contribution Guidelines

### For Members

Once you are an approved member, please follow these guidelines:

#### Code Contributions

1. **Fork and Branch:** Create a feature branch from `main`
   ```bash
   git checkout -b feature/your-feature-name
   ```

2. **Code Standards:**
   - Follow PEP 8 for Python code
   - Use meaningful variable and function names
   - Add docstrings to all functions and classes
   - Include type hints where applicable

3. **Testing:**
   - Write unit tests for all new functionality
   - Ensure all tests pass before submitting a PR
   - Aim for >80% code coverage

4. **Documentation:**
   - Update relevant documentation
   - Add comments for complex logic
   - Update README if adding new features

5. **Pull Requests:**
   - Use the PR template
   - Provide clear description of changes
   - Reference related issues
   - Request review from at least one core team member

#### Research Contributions

1. **Benchmarks:**
   - Document methodology clearly
   - Provide reproducible scripts
   - Include data sources and licensing information
   - Add results to the appropriate directory

2. **Analysis:**
   - Use Jupyter notebooks for exploratory analysis
   - Export final results to reproducible scripts
   - Document all assumptions and limitations

3. **Publications:**
   - Coordinate with team on authorship
   - Follow Columbia's research integrity guidelines
   - Acknowledge all contributors appropriately

### Code Review Process

- All contributions require peer review
- At least one core team member must approve
- Automated tests must pass
- Code must meet quality standards
- Reviewer feedback must be addressed

### Commit Messages

Use clear, descriptive commit messages:
```
[Category] Brief description

Longer explanation if needed

- Bullet points for multiple changes
- Reference issues: Fixes #123
```

Categories: `[Feature]`, `[Bugfix]`, `[Docs]`, `[Test]`, `[Refactor]`

## ü§ù Collaboration Etiquette

- **Be Respectful:** Treat all members with courtesy and professionalism
- **Be Constructive:** Provide helpful feedback in code reviews
- **Be Responsive:** Reply to comments and requests in a timely manner
- **Be Transparent:** Share your work and communicate blockers
- **Be Inclusive:** Welcome diverse perspectives and approaches

## üìä Authorship and Attribution

- **Code Contributions:** Tracked through Git history
- **Research Contributions:** Discussed and agreed upon by contributors
- **Publications:** Follow academic standards for authorship
- **Credit:** All contributors will be acknowledged

### Publication Policy

For papers and publications arising from this work:
1. Authorship is determined by intellectual contribution
2. Follow Columbia's authorship guidelines
3. Discuss authorship early in the research process
4. All co-authors must approve before submission

## üîê Confidentiality

- **Research Data:** May be subject to confidentiality agreements
- **Internal Discussions:** Keep internal communications within the team
- **Public Sharing:** Coordinate with team before sharing work externally
- **Academic Integrity:** Follow Columbia's policies on data sharing and publications
